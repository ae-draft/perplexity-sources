# 📊 Промпт для анализа спринтов и нагрузки команды

**Версия:** 1.1  
**Последнее обновление:** 10 марта 2026 г.  
**Источник истины:** `department.md`, `reference.md`, `templates.md`, `analysis-pmt.md`

---

## Цель

Провести анализ задач из выгрузки JIRA (CSV/XML) для оценки успешности спринтов, планирования новых спринтов, анализа рисков и нагрузки команды с использованием данных из `reference.md`.

---

## Типы анализа

### 1. Анализ завершённого спринта (Sprint Review)

**Назначение:** Оценка успешности выполнения спринта после его завершения.

**Входные данные:**
- Выгрузка JIRA (CSV/XML)
- Название завершённого спринта
- Дата окончания спринта

**Ключевые метрики:**

| Метрика | Формула | Источник |
|---------|---------|----------|
| План vs Факт | (Завершено задач / Запланировано задач) × 100% | Статусы задач |
| Внеплановые задачи | Задачи, добавленные после начала спринта | Дата создания vs Дата начала спринта |
| Невыполненные задачи | Задачи в статусах "В разработке", "На паузе", "К доработке" | Статусы задач |
| Причины невыполнения | Метки, комментарии, блокеры | Поля задачи |

**Алгоритм:**
1. Найти все задачи в спринте по названию спринта
2. Отфильтровать по статусам (Завершено/Слит vs Активные)
3. Выявить внеплановые задачи (дата создания > даты начала спринта)
4. Сгруппировать невыполненные задачи по причинам
5. Рассчитать метрики успешности

**Формула успешности спринта:**
```
Успешность (%) = (Завершено задач / Запланировано задач) × 100
```

**Уровни успешности:**
| Процент | Уровень | Иконка |
|---------|---------|--------|
| >= 90% | Отлично | 🟢 |
| 70-89% | Хорошо | 🟡 |
| 50-69% | Удовлетворительно | 🟠 |
| < 50% | Критично | 🔴 |

### 2. Планирование нового спринта (Sprint Planning)

**Назначение:** Оценка нагрузки на команду и соотношения трудоемкости задач и мощности команды.

**Входные данные:**
- Выгрузка JIRA (CSV/XML)
- Название планируемого спринта
- Состав команды (из `department.md`)
- Нормы нагрузки (из `reference.md` Раздел 4)

**Ключевые метрики:**

| Метрика | Формула | Источник |
|---------|---------|----------|
| Мощность команды | Сумма норм часов по всем сотрудникам | `reference.md` Раздел 4 |
| Трудоемкость спринта | Сумма оценок всех задач в спринте | Оценка разработки (часы) |
| Загрузка команды | (Трудоемкость / Мощность) × 100% | Расчёт |
| Резерв | Мощность - Трудоемкость | Расчёт |

**Алгоритм:**
1. Получить состав команды из `department.md` по названию спринта
2. Получить нормы часов для каждой роли из `reference.md`
3. Рассчитать общую мощность команды (часы/неделю)
4. Суммировать оценку разработки по всем задачам спринта
5. Рассчитать загрузку и резерв
6. Сформировать рекомендации

**Формулы:**
```
Мощность команды = Σ(Норма часов по роли для каждого сотрудника)
Трудоемкость спринта = Σ(Оценка разработки по всем задачам)
Загрузка (%) = (Трудоемкость / Мощность) × 100
Резерв (часы) = Мощность - Трудоемкость
```

**Рекомендации по загрузке:**
| Загрузка | Статус | Рекомендация |
|----------|--------|--------------|
| <= 80% | 🟢 Норма | Можно добавить задачи |
| 81-95% | 🟡 Оптимально | Спринт сбалансирован |
| 96-110% | 🟠 Риск переработки | Рассмотреть удаление задач |
| > 110% | 🔴 Критично | Требуется перепланирование |

### 3. Анализ рисков

**Назначение:** Выявление и оценка рисков по задачам спринта.

**Входные данные:**
- Выгрузка JIRA (CSV/XML)
- Метрики рисков из `reference.md` (Раздел 3)

**Факторы риска (из reference.md):**

| Фактор риска | Вес | Порог |
|--------------|-----|-------|
| Задача на паузе > 7 дней | 3 | Критический |
| Блокер без спринта | 2 | Высокий |
| FeedbackPending > 5 дней | 2 | Высокий |
| Нет исполнителя | 1 | Средний |
| Просрочен дедлайн | 3 | Критический |

**Алгоритм:**
1. Для каждой задачи проверить факторы риска
2. Рассчитать вес риска: `Сумма весов активных факторов`
3. Определить уровень риска по порогу

**Формула расчёта уровня риска:**
```
Если Вес >= 3 → 🔴 Критический
Если Вес >= 2 → 🟠 Высокий
Если Вес >= 1 → 🟡 Средний
Если Вес = 0 → 🟢 Низкий
```

---

### 4. Анализ нагрузки по сотрудникам

**Назначение:** Оценка фактической нагрузки сотрудников и выявления переработок.

**Входные данные:**
- Выгрузка JIRA (CSV/XML)
- Нормы нагрузки из `reference.md` (Раздел 4)
- Период анализа (спринт или диапазон дат)

**Алгоритм:**
1. Сгруппировать задачи по исполнителям
2. Суммировать оценку разработки по каждому исполнителю
3. Получить роль каждого сотрудника из `reference.md`
4. Сравнить с нормой часов/неделю из справочника
5. Выявить переработки и недогрузки

**Формулы:**
```
Загрузка (%) = (Фактические часы / Норма часов) × 100
Переработка (часы) = Фактические часы - Норма часов
```

**Статусы нагрузки:**
| Загрузка | Статус | Иконка |
|----------|--------|--------|
| < 80% | Недогрузка | 🔵 |
| 80-100% | Норма | 🟢 |
| 101-110% | Легкая переработка | 🟡 |
| > 110% | Критическая переработка | 🔴 |

## Критически важные правила извлечения данных

### 1. СТРОГОЕ следование данным из источника

| Правило | Требование |
|---------|------------|
| ❌ ЗАПРЕЩЕНО | Придумывать, интерполировать или додумывать любые данные |

2. Прочитать значение названия спринта

| Название спринта | Команда |
|------------------|---------|
| "Команда Локиса ДД.ММ-ДД.ММ" | Команда Локиса |
| "Команда Батукаева ДД.ММ-ДД.ММ" | Команда Батукаева |
| "Команда Овчинникова ДД.ММ-ДД.ММ" | Команда Овчинникова |
| "Команда OLAP ДД.ММ-ДД.ММ" | Команда OLAP |

| Статус | Включать в анализ |
|--------|-------------------|
| Разработка | ✅ |
| Ревью | ✅ |
| К разработке | ✅ |
| К доработке | ✅ |

| Слит | ✅ (для анализа завершённого спринта) |

## Формат отчёта об анализе

### Формат точности оценок

**Формула:** `Точность = (Факт / Оценка) × 100%`

| Пример | Оценка | Факт | Точность | Отклонение | Статус |
|--------|--------|------|----------|------------|--------|
| FRONT-3144 | 8.0 ч | 20.0 ч | 250% | +150% | 🔴 Перерасход |
| FRONT-3189 | 8.0 ч | 8.0 ч | 100% | 0% | 🟢 Точно |
| BACK-2240 | 9.0 ч | 2.5 ч | 28% | -72% | 🟢 Экономия |

**Как читать:**
- **>100%** — делали ДОЛЬШЕ (перерасход)
- **<100%** — сделали БЫСТРЕЕ (экономия)
- **=100%** — точно в оценку

**Рекомендуемый формат вывода:**
```
| Задача | Оценка | Факт | Точность | Отклонение | Статус |
|--------|--------|------|----------|------------|--------|
| FRONT-X | 8.0 ч | 20.0 ч | 250% | +150% | 🔴 Перерасход |
```

### Группировка по компонентам

**Дополнительно к типам задач** группировать по компонентам системы:

| Компонент | Примеры задач |
|-----------|---------------|
| **Таблицы** | FRONT-3106, FRONT-3104, FRONT-3077 |
| **Карточка проекта** | FRONT-3184, FRONT-3183, FRONT-3093 |

| **Клиентские дефекты (Win)** | DEFECT-438, DEFECT-425, DEFECT-418 |

**Формат вывода:**
```
| Компонент | Задач | Оценка (ч) | Факт (ч) | Точность | Коэффициент |
|-----------|-------|------------|----------|----------|-------------|
| Таблицы | 5 | 45.0 | 180.0 | 400% | ×4.0 |
| Карточка проекта | 8 | 60.0 | 150.0 | 250% | ×2.5 |
| Отчёты | 4 | 30.0 | 45.0 | 150% | ×1.5 |
```

### Формат причин невыполнения

**Источники данных:**
1. **Статус задачи** — поле `Статус`
2. **Комментарии** — поле `Комментарий`
3. **Блокеры** — поле `Входящая связь задачи (Блокирует)`
4. **Причина паузы** — поле `Пользовательское поле (Причина перевода на паузу)`

**Алгоритм формирования:**
```
1. Если Статус = "На паузе":
   → Ищем в Комментариях причину
   → Если есть "FeedbackPending" → "FeedbackPending — требуется уточнение у [Автор]"
   → Если есть "Внешний" → "Внешний блокер — [Описание]"

2. Если Статус = "К доработке":
   → "Требуется исправление — [Комментарий тестировщика]"

3. Если Статус = "Анализ":
   → "Задача на этапе исследования — оценка не определена"

4. Если есть Блокеры:
   → "Блокируется задачей [KEY] — [Статус блокера]"

5. Если данных нет:
   → "⚠️ Причина не указана в данных"
```

---

### Формат сравнения спринтов

**Формула:** `Изменение = ((Новое - Старое) / Старое) × 100%`

| Метрика | Спринт 1 | Спринт 2 | Изменение | Статус |
|---------|----------|----------|-----------|--------|
| **Точность оценок** | 176% | 136% | -40% | ✅ Улучшение |
| **Завершено задач** | 6 | 11 | +83% | ✅ Улучшение |
| **Успешность спринта** | 75% | 79% | +4% | ✅ Улучшение |

**Как читать:**
- **Для точности оценок:** Меньше = Лучше (176% → 136% = ✅ улучшение)
- **Для завершённых задач:** Больше = Лучше (6 → 11 = ✅ улучшение)
- **Отрицательное изменение точности** = Хорошо (меньше перерасход)
- **Положительное изменение задач** = Хорошо (больше выполнено)

## Важные замечания

### О расчётах

| ❌ НЕПРАВИЛЬНО | ✅ ПРАВИЛЬНО |
|---------------|-------------|
| Округлять коэффициенты до целых | Использовать точные значения из `reference.md` |
| Игнорировать задачи без исполнителя | Включать в анализ с пометкой "Не назначен" |
| Считать переработку от фактических часов | Считать переработку от нормы часов по роли |

### О рисках

| ❌ НЕПРАВИЛЬНО | ✅ ПРАВИЛЬНО |
|---------------|-------------|
| Придумывать факторы риска | Использовать только факторы из `reference.md` |
| Игнорировать задачи на паузе | Проверять длительность паузы (> 7 дней = критический) |
| Не проверять спринт блокера | Проверять наличие спринта у каждого блокера |

### О нагрузке

| ❌ НЕПРАВИЛЬНО | ✅ ПРАВИЛЬНО |
|---------------|-------------|
| Сравнивать с универсальной нормой 40 часов | Использовать норму по роли из `reference.md` |
| Игнорировать роли сотрудников | Определять роль по `reference.md` или `department.md` |
| Считать только текущий спринт | Учитывать все активные задачи сотрудника |

### О завершённых спринтах

| ❌ НЕПРАВИЛЬНО | ✅ ПРАВИЛЬНО |
|---------------|-------------|
| Считать только задачи со статусом "Завершено" | Включать "Слит" и "Готово" как завершённые |
| Игнорировать внеплановые задачи | Выделять задачи, добавленные после начала спринта |
