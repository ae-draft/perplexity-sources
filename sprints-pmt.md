# 📋 Промпт для построения отчета по спринтам JIRA

**Версия:** 3.1  
**Последнее обновление:** 22 февраля 2026 г.  
**Источник истины:** `department.md`, `reference.md`, `templates.md`, `sprints-pmt.md`

---

## Цель

Построить отчет по активным задачам из выгрузки JIRA (CSV/XML), сгруппированных по спринтам и статусам.

---

## Критически важные правила извлечения данных

### 1. СТРОГОЕ следование данным из источника

| Правило | Требование |
|---------|------------|
| ❌ ЗАПРЕЩЕНО | Придумывать, интерполировать или додумывать любые данные |
| ✅ ОБЯЗАТЕЛЬНО | Брать ТОЧНЫЕ значения из полей CSV/XML |
| ⚠️ Если данные отсутствуют | Писать "Не указано", а НЕ придумывать значения |
| 👤 При указании ФИО | Брать ТОЛЬКО из поля `Исполнитель` или `displayname` |

### 2. Определение команды по спринту

> ⚠️ **КРИТИЧЕСКАЯ ОШИБКА:** Команда определяется ТОЛЬКО по названию спринта, НЕ по отв. разработчику!

**Алгоритм определения команды:**

1. Найти поле `Спринт` (CSV) или тег `<customfield id="customfield_10101">` (XML)
2. Прочитать значение названия спринта
3. Определить команду по НАЗВАНИЮ спринта:

| Название спринта | Команда |
|------------------|---------|
| "Команда Локиса ДД.ММ-ДД.ММ" | Команда Локиса |
| "Команда Батукаева ДД.ММ-ДД.ММ" | Команда Батукаева |
| "Команда Овчинникова ДД.ММ-ДД.ММ" | Команда Овчинникова |
| "Команда OLAP ДД.ММ-ДД.ММ" | Команда OLAP |

> ⚠️ **НЕ ПУТАТЬ:** Отв. разработчик ≠ Команда (название спринта)

### 3. Точное извлечение полей

#### Исполнитель (Assignee)
→ Брать ТОЧНО как в поле

Если исполнитель не назначен:
→ Писать: **"Не назначен"**

#### Отв. разработчик
→ Брать username из соответствующего поля

#### Приемка/Тестировщик
**Приоритет полей (в порядке убывания):**
1. `Отв. тестировщик` — если есть
2. `Отв. аналитик` — если нет тестировщика
3. `Автор задачи` — если нет предыдущих

#### Статус задачи

| Статус | Раздел отчета |
|--------|---------------|
| Разработка | В разработке |
| Ревью | В ревью |
| К разработке | К разработке |
| К доработке | К доработке |
| На паузе | На паузе |
| Анализ | ИСКЛЮЧИТЬ |
| Завершено | ИСКЛЮЧИТЬ |
| Готово | ИСКЛЮЧИТЬ |
| Слит | ИСКЛЮЧИТЬ |
| Отменён | ИСКЛЮЧИТЬ |

#### Оценка и время

| Поле | Формат |
|------|--------|
| Оценка разработки (часы) | `16.0` часов |
| Потрачено (время) | Из поля `Затраченное время` (конвертация секунд: 86400 сек = 1 день, 3600 сек = 1 час) |
| Трудоемкость | Указывать только если значение > 0 |

---

### 4. Фильтрация задач для отчета

#### ✅ ВКЛЮЧАТЬ в отчет:
- Статусы: **В разработке, Ревью, К разработке, К доработке, На паузе**
- Спринты: текущие спринты периода

#### ❌ ИСКЛЮЧАТЬ из отчета:
- Статусы: **Завершено, Готово, Отменён, Слит, Анализ**
- Задачи вне указанных спринтов

---

### 5. Проверка полноты данных

**ОБЯЗАТЕЛЬНАЯ проверка перед формированием отчета:**

Для каждого спринта и каждого статуса:
1. Выполнить поиск по всем задачам с данным статусом
2. Составить полный список `task_key` всех найденных задач
3. Сверить список с задачами в отчете
4. Если задача пропущена → добавить в отчет

**Особое внимание:**
> ⚠️ Статус **"На паузе"** — часто пропускается!  
> ⚠️ Задачи без исполнителя  
> ⚠️ Задачи с несколькими спринтами — брать **ПОСЛЕДНИЙ** спринт из списка

---

### 6. Обработка блокеров

#### Правила обработки блокеров:

| Ситуация | Формулировка |
|----------|--------------|
| Блокер завершён (Слит/Завершено) | `✅ Слит (ДД.ММ.ГГГГ). Блокер завершён, задача готова к приёмке.` |
| Блокер в работе | `⏳ Активен — [статус]. Задача ожидает решения блокера.` |
| Блокер на паузе | `⏳ На паузе — [причина из данных].` |
| Внешний блокер (дампы, логи, клиент) | `⚠️ Внешний блокер — [что ожидается]. Задача не может быть продолжена без данных.` |
| Метка FeedbackPending | `⚠️ FeedbackPending — задача на паузе до получения обратной связи. Точная причина паузы в данных не указана. Требуется уточнение у [Автор задачи].` |
| Дочерние задачи блокируют | `⚠️ Дочерние задачи — [список]. Задача переведена на паузу до исправления всех дочерних дефектов.` |

#### Проверка спринта блокера:

1. Найти спринт задачи-блокера в данных
2. Если спринт указан с датами:
   - **Прошедший спринт** → `✅ Завершён`
   - **Текущий спринт** → `⏳ В текущем спринте`
   - **Будущий спринт** → `📅 Запланирован в следующий спринт`
3. Если спринт **не указан**:
   - `⚠️ Спринт не указан — задача не запланирована. Требуется приоритизация.`

### 8. Алгоритм работы

| Шаг | Действие |
|-----|----------|
| **Шаг 1** | Загрузка и парсинг данных (CSV/XML) |
| **Шаг 2** | Извлечение списка всех спринтов текущего периода |
| **Шаг 3** | Группировка задач по спринтам (по НАЗВАНИЮ спринта) |
| **Шаг 4** | Группировка задач по статусам внутри каждого спринта |
| **Шаг 5** | Извлечение данных для каждой задачи (строго из источника) |
| **Шаг 6** | Обработка блокеров (проверка спринта, статуса) |
| **Шаг 7** | Проверка полноты (сверка количества задач) |
| **Шаг 8** | Формирование сводки по командам и статусам |
| **Шаг 9** | Формирование таблицы блокеров |
| **Шаг 10** | Вывод финального отчета в формате markdown |

### 9. Контрольный чек-лист перед выводом отчета

- [ ] Все команды определены по **НАЗВАНИЮ спринта**, не по отв. разработчику
- [ ] Все ФИО взяты **ТОЧНО из данных**, без додумывания отчеств
- [ ] Проверено наличие задач в статусе **"На паузе"**
- [ ] Все задачи без исполнителя помечены как **"Не назначен"**
- [ ] Количество задач в сводке **совпадает с фактическим в данных**
- [ ] Нет задач в статусе **"Завершено", "Готово", "Слит", "Анализ"**
- [ ] Все ссылки на задачи корректны: `https://jira.a2nta.ru/browse/[KEY]`
- [ ] Проверены все спринты периода
- [ ] Проверены все статусы (В разработке, Ревью, К разработке, К доработке, На паузе)
- [ ] **Блокеры:** проверен спринт каждого блокера, указан статус планирования
- [ ] **Блокеры:** нет интерпретаций причин паузы без явных данных
- [ ] **Блокеры:** для FeedbackPending указано "Требуется уточнение у [Автор]"
- [ ] **Справочник:** все ФИО сверены с `reference.md`
- [ ] **Шаблоны:** структура отчёта взята из `templates.md`
